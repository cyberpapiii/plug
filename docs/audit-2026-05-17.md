# Plug Audit: MCP Spec + Rust Stack Refresh

Date: 2026-05-17

Scope: research and planning only. No code changes were made for this audit.

## 1. Current State

### Product shape

Plug is a Rust MCP multiplexer. Downstream clients connect to Plug once, and Plug fans out to multiple upstream MCP servers while preserving a single routed catalog.

Current repo truth says the roadmap is complete on `main` for the existing production-ready bar. The active surfaces are:

- `plug connect`: stdio downstream adapter for local AI clients.
- `plug start` / `plug serve --daemon`: shared background service owning daemon IPC plus downstream HTTP/HTTPS.
- `plug serve`: currently delegates to daemon mode rather than being a meaningfully separate standalone foreground runtime path in `plug/src/runtime.rs`.
- `plug-core::Engine`: shared runtime truth, config snapshot, server registry, routing, event bus, health, reconnect, reload, and shutdown coordination.
- `ServerManager`: upstream lifecycle, health, circuit breakers, semaphores, auth state, reconnect.
- `ToolRouter`: merged downstream-facing MCP surface, name routing, capability synthesis, lazy discovery, tasks, artifacts, notification routing, roots, subscriptions, and reverse-request bridges.
- `HttpState` / `SessionStore`: downstream Streamable HTTP sessions and SSE delivery.
- `daemon.rs` / `ipc_proxy.rs`: daemon-backed stdio proxy, live-session inventory, IPC control plane, and reverse-request forwarding.

### Auth model

Upstream:

- None, bearer token, and OAuth are represented in config.
- OAuth 2.1 + PKCE upstream auth is implemented with metadata discovery, auth-code exchange, credential storage, token refresh, CLI login/complete/status, injected token support, and daemon auth-state notification.
- Scope configuration exists per upstream server through `oauth_scopes`.

Downstream:

- `auto`: loopback means no bearer auth; non-loopback means bearer auth.
- `bearer`: explicit bearer auth.
- `oauth`: local downstream OAuth authorization-code + PKCE support, authorization server metadata endpoint, access/refresh token issuance, token refresh, and protected `/mcp` access.
- Static configured downstream client ID/secret is supported. I did not find dynamic client registration.

Security controls already present:

- Non-loopback HTTP cannot run with `auth_mode = "none"`.
- Non-loopback HTTP requires TLS cert/key.
- TLS private key must not be group/world-readable.
- Origin validation exists for DNS rebinding protection, with explicit allowlist and loopback matching.
- HTTP body limit is 4 MB.
- Bearer token verification uses constant-time comparison.
- Legacy SSE upstream transport has SSRF-oriented endpoint validation.

### Multiplexing strategy

- Upstream discovery/registration is config-driven.
- `ServerManager` starts enabled servers in bounded batches, tracks health, auth, circuit breakers, reconnect flags, and semaphores.
- `ToolRouter::refresh_tools()` builds one immutable routed snapshot with tools/resources/prompts/routes.
- Tool names are always prefixed in v0.1 using `ServerName__tool_name`, with optional `tool_groups` and `tool_renames`.
- Name collisions are avoided through mandatory prefixing rather than runtime collision negotiation.
- Per-client catalog shaping exists:
  - Windsurf 100-tool cap.
  - VS Code Copilot 128-tool cap.
  - Claude/Cursor/Codex native lazy mode.
  - OpenCode bridge mode.
- Lazy bridge mode exposes `plug__search_tools`, then search loads a bounded session working set of real routed tool names. Direct routed calls are used after loading.
- Legacy `meta_tool_mode = true` remains as deprecated compatibility and exposes `plug__list_servers`, `plug__list_tools`, `plug__search_tools`, and `plug__invoke_tool`.
- Error propagation generally converts upstream errors to MCP errors while preserving transport/session failures enough to trigger reconnect and retry.
- Large results can spill to synthetic `plug://artifact/...` resources.

### Non-spec extensions / custom behavior

- `plug__search_tools` bridge discovery.
- Legacy meta-tools, including `plug__invoke_tool`.
- `plug://artifact/...` resource manifests/chunk reads for oversized results.
- Operator HTTP endpoint `/_plug/live-sessions` protected by `x-plug-operator-token`.
- Daemon IPC protocol versioning (`IPC_PROTOCOL_VERSION = 3`).
- Client-specific lazy policy and tool-limit filtering.
- Tool metadata enrichment/inference.

Current published protocol version is `2025-11-25`. Plug is already aligned on the advertised protocol string and implements most of the high-value multiplexer primitives from that spec: Streamable HTTP, structured tool output, resource links, elicitation, sampling, roots, completion, OAuth, and tasks.

The biggest spec-shaped gaps are not “missing tools/prompts/resources.” They are multiplexer edge cases:

- **SSE resumability**: the spec describes event IDs plus `Last-Event-ID` replay. Plug emits SSE IDs but appears not to persist/replay missed messages. For a single-server implementation this is optional; for a multiplexer, dropped list-changed/progress/reverse-request notifications can leave downstream clients stale.
- **Resource subscription parity**: HTTP supports subscribe/unsubscribe, but daemon IPC/stdout clients do not. Since many local clients still use stdio, Plug’s advertised capabilities must stay transport-masked correctly.
- **Dynamic client registration**: downstream OAuth appears static-client only. The MCP authorization model and modern remote-connector flows increasingly assume DCR or at least richer metadata.
- **Server identity / server cards**: Plug has `/.well-known/mcp.json`, but this needs a fresh check against current server-card expectations and registry conventions.
- **Protocol negotiation strictness**: Plug requires exact `2025-11-25` on post-initialize HTTP. The local docs say older versions should be accepted with reduced feature set, but the code rejects mismatched versions. That may be fine for a strict product stance, but it is a doc/behavior mismatch and can hurt older clients.

Ambiguous / ecosystem-not-converged:

- Tasks are present in the `2025-11-25` docs but still newer and less universally supported by clients.
- Server cards, registry discovery, and future stateless/session-elevation ideas are still moving. Do not build a large custom abstraction before checking current SEP status.

### Security and ops

Sources:

- MCP security best practices: https://modelcontextprotocol.io/specification/2025-11-25/basic/security_best_practices
- MCP authorization: https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization
- MCP transports security guidance: https://modelcontextprotocol.io/specification/2025-11-25/basic/transports
- OWASP Top 10 for LLM Applications: https://owasp.org/www-project-top-10-for-large-language-model-applications/
- MCP Registry: https://github.com/modelcontextprotocol/registry

Plug is already doing several important things right for a multiplexer:

- Namespacing tool names by server avoids basic collision and tool-poisoning ambiguity.
- Origin validation and loopback default reduce DNS rebinding exposure.
- Non-loopback serving requires auth and TLS.
- Secrets are redacted in debug output in the SSE client path.
- Tool annotations and display metadata are normalized.
- Operator endpoints have a separate operator token.

Gaps that matter specifically because Plug aggregates tools:

- **Tool poisoning / identity**: prefixing is good, but Plug should preserve and surface upstream server identity, source URL/command, and trust tier in machine-readable inventory. A multiplexer can otherwise make untrusted tools look first-party.
- **Approval context**: destructive/open-world/idempotent hints are enriched, but downstream clients vary in how much they honor them. Plug should produce an operator-facing “risky tool inventory” and avoid over-inference for destructive tools.
- **Sandboxing stdio upstreams**: Plug spawns arbitrary commands from config. There is no clear sandbox/profile/allowlist model for stdio upstream processes. That may be acceptable for local developer use, but it should be explicit in docs and optionally enforceable.
- **OAuth confused deputy**: aggregating upstream OAuth servers and serving downstream OAuth creates two auth planes. Scope display, token provenance, and auth-state reporting should make it impossible to confuse downstream client authorization with upstream server authorization.
- **Trace correlation**: Plug has structured logging and call IDs, but no OpenTelemetry propagation/export. For proxy-boundary incidents, trace IDs across downstream request -> router -> upstream call -> retry/reconnect would materially improve diagnosis.

## 4. Prioritized Roadmap

### Do now

1. Replace `serde_yml`.
   - Candidate replacements from RustSec: `serde_norway` or `serde_yaml_ng`.
   - First step: find every `serde_yml` call site and identify whether YAML is user-facing, test-only, or config import/export.
   - Keep output compatibility if users consume YAML.

2. Decide and document protocol-version compatibility.
   - Option A: strict `2025-11-25` only after initialize. Update docs and tests to make this explicit.
   - Option B: accept older headers with reduced capability masks. More compatibility work, more test matrix.
   - Recommendation: Option A for now unless a real client needs older post-initialize headers.
   - Decoupled cleanup: keep the `2025-11-25` behavior, but replace the HTTP initialize-body JSON mutation with typed `InitializeResult::with_protocol_version(...)` in `plug-core/src/http/server.rs`; this can ship before the `rmcp 1.7.0` bump.

3. Upgrade `rmcp 1.5.0 -> 1.7.0` with prerequisites, not a broad refactor.
   - Prerequisites: keep Plug's custom HTTP server in place, update manifests/lockfile only, then fix compile/test fallout from SDK type changes.
   - Add parity checks for Streamable HTTP Origin/Host validation, HTTP/2 `:authority`, init timeout, stdio parse-error behavior, and prompt resource-content shape.
   - Do not adopt `rmcp`'s Streamable HTTP server/session-store stack in this same change; evaluate that separately after protocol-version and resumability decisions.
   - Run serialized workspace tests: `cargo test --workspace -- --test-threads=1`.

### Do soon

4. Close the resource-subscribe transport gap.
   - Option A: implement daemon IPC subscribe/unsubscribe and targeted `ResourceUpdated` delivery.
   - Option B: keep unsupported but add stronger tests proving daemon/stdout capability masking.
   - Recommendation: Option B first, then A only if stdio clients actually need subscriptions.

5. Add downstream SSE replay or explicitly document non-resumable behavior.
   - If implementing, keep a bounded per-session event buffer keyed by event ID.
   - Include list_changed/progress/resource update/reverse request behavior in tests.

6. Harden remote/downstream OAuth by following active auth SEPs, not inventing Plug-specific client-registration semantics.
   - SEP-991 makes URL-based client metadata documents the preferred path and DCR optional; SEP-1046, SEP-990, SEP-985, and SEP-2207 already define accepted/final auth direction; SEP-1932 and SEP-1933 are draft extension work for DPoP and workload identity.
   - Add DCR only when implementing an official profile or when a real client requires it; otherwise improve metadata, scope display, and error messages.
   - Because the current public surface is personal-public/solo, a one-off private connector workaround is acceptable if hidden behind config and explicitly not treated as a stable public contract.

7. Address HTTPS serving transitive advisory.
   - Investigate replacing `axum-server` or loading PEM with `rustls-pki-types` before handing config to the server stack.

### Consider later

8. Add operator trust inventory.
   - Machine-readable source/trust metadata per server/tool.
   - Risky tool report based on upstream annotations plus local inference confidence.

9. Add OpenTelemetry-style trace correlation aligned with SEP-2243 and enterprise roadmap language.
   - Trace/span ID across downstream request, routed call, upstream retry, auth refresh, and artifact spillover.

10. Add optional stdio sandboxing.
   - Useful for untrusted upstreams, but likely too disruptive as a default.

11. Revisit WebSocket only if a real client/server dependency appears.
   - The 2026 roadmap says no additional official transports are planned this cycle.

## 5. Open Decisions

1. Protocol compatibility stance:
   - **Recommended**: strict `2025-11-25`, update docs to match.
   - Alternative: support older protocol headers with explicit transport/capability downgrades.
   - Re-review when the `2026-06-30` stateless/per-request capability work lands in SDKs.
   - Follow-up from Section 8: delete the manual HTTP initialize JSON rewrite after moving the version onto the typed `InitializeResult`; keep hard-coded constants until a central protocol-version module replaces them.

2. Resource subscription parity:
   - **Recommended**: test and document daemon IPC masking now; implement IPC parity only if local stdio clients need it.
   - Alternative: implement full IPC subscription delivery immediately.
   - SEP-2549 TTLs may reduce the practical need for subscriptions, but do not eliminate Plug's masking responsibility.
   - Public-surface weighting: current evidence does not show external users depending on subscribe parity, so this can remain deferred unless Rob's local stdio clients need it.

3. Downstream OAuth DCR:
   - **Recommended**: follow SEP-991's URL-based client metadata direction, the accepted/final auth SEPs, and draft DPoP/workload-identity extensions; do not design a Plug-specific public DCR variant.
   - Alternative: implement DCR proactively for ecosystem alignment.
   - Public-surface weighting: because Plug is not currently a high-commitment distributed product, a private one-client auth workaround is acceptable if isolated, but it should not displace the official profiles in docs or defaults.

4. YAML replacement:
   - **Recommended**: choose the maintained fork closest to current behavior after call-site inventory.
   - Alternative: remove YAML support if it is nonessential.

5. `rmcp` ownership boundary:
   - **Recommended**: upgrade to `1.7.0` now, but keep Plug-owned routing/lazy discovery/task ownership because those are multiplexer-specific.
   - Alternative: keep local implementation stable and upgrade SDK only for security/compat.
   - Separate decision: whether to later replace Plug's custom downstream HTTP server with `rmcp`'s Streamable HTTP server plus custom layers.

6. Security posture for stdio upstreams:
   - **Recommended**: document trusted-local-config assumption now; consider opt-in sandboxing later.
   - Alternative: design a sandbox policy immediately, with higher implementation and support cost.

7. Tasks migration:
   - **Recommended**: maintain `2025-11-25` task compatibility for current clients, but do not expand Plug's current core-task implementation until SEP-2663 (`io.modelcontextprotocol/tasks`) is reconciled.
   - Alternative: begin an extension-mode tasks implementation now, with higher churn risk before SDK support lands.
   - Public-surface weighting: current task support can remain compatibility-only longer; no evidence shows external clients depending on deeper Plug-specific task semantics.

8. Custom lazy bridge future:
   - **Recommended**: keep `plug__search_tools` / `plug__invoke_tool` as Plug-owned client adaptation. No active SEP found that replaces lazy tool discovery directly.
   - Alternative: deprecate legacy meta-tool mode once clients converge on a native discovery/search pattern.
   - Public-surface weighting: this is a real local-client adaptation, not a public ecosystem promise; preserve the useful bridge while avoiding extra compatibility shims for speculative clients.

### Roadmap priority areas

**Transport evolution and scalability**: The roadmap explicitly says the project is not adding more official transports this cycle. The work is instead to make Streamable HTTP operationally scalable: stateless operation, session/resumption semantics, proxy/load-balancer behavior, and server-card discovery. For Plug, this means WebSocket should stay deferred, while HTTP header standardization, stateless/sessionless design, and resumability deserve priority because Plug is itself a gateway/proxy. Plug already has Streamable HTTP downstream and upstream support, session IDs, and basic SSE reconnect behavior, but it hand-rolls strict protocol-version handling, session state, reverse-request routing, and notification fan-out.

**Agent communication**: The roadmap names Tasks as the main agent-communication primitive, then points to retry and expiry semantics as unresolved production gaps. Since the roadmap, SEP-2663 moved tasks into an official extension (`io.modelcontextprotocol/tasks`) and explicitly redesigns the `2025-11-25` core task feature. Plug already implements `2025-11-25` task wrapping, task ownership, artifact spillover, upstream task passthrough, and stdio/HTTP task APIs. That work should remain compatibility support, but Plug should not deepen the current task model until the extension shape lands in SDKs.

**Governance maturation**: The project has formalized WGs/IGs, contributor ladder, extension governance, and SDK tiering. This does not obsolete Plug code, but it changes how to time upgrades: accepted/final SEPs plus SDK WG rollout matter more than ad hoc community patterns. For Plug, the most relevant governance output is extension mechanics and SDK tiering, because future Tasks, Apps, auth, and transport behavior will likely arrive as extensions before becoming core.

**Enterprise readiness**: The roadmap calls out audit trails, observability, enterprise-managed auth, gateway/proxy behavior, and configuration portability. This is the strongest signal that Plug-owned work remains valuable. The spec may standardize auth profiles, headers, server cards, and interceptor primitives, but it will not build Plug's cross-upstream provenance, routing, trace correlation, capability synthesis, or policy surface.

### Risk assessment

**Downstream MCP clients**: Low if the upgrade is limited to `rmcp` and compile fixes. Wire-level behavior should remain Plug-controlled because downstream HTTP and IPC stay local. Risk becomes medium/high only if Plug adopts `rmcp`'s Streamable HTTP server in the same change, because session IDs, Origin/Host validation, SSE reconnects, and error shapes can change.

**Plug config format**: No expected breaking change from `rmcp 1.7.0`. Server config, OAuth config, lazy-tool config, and IPC protocol are Plug-owned.

**Test surface to update**:

- Full serialized workspace run: `cargo test --workspace -- --test-threads=1`.
- HTTP transport tests for `MCP-Protocol-Version`, `Mcp-Session-Id`, Origin/Host handling, HTTP/2 authority fallback if practical, and init timeout.
- Task tests covering local task wrapping, upstream task passthrough, task artifact spillover, stdio daemon-backed tasks, and session replacement.
- Reverse request tests for elicitation/sampling over stdio, HTTP SSE, and daemon IPC.
- Prompt/resource-content tests only if compile errors or schema snapshots show the flattened `PromptMessageContent::Resource` shape affects Plug.

**Control to preserve**:

### Decision

**Upgrade with prerequisites.** Move to `rmcp 1.7.0` now for SDK bugfixes and current ecosystem alignment, but keep the change narrow:

### Blame and SDK finding

`git blame` traces the manual body rewrite to commit `7927f56` (`fix(http): return 2025-11-25 in initialize response body`, 2026-03-10). The paired bug report, `docs/bug-reports/claude-remote-mcp-no-tools-when-initialize-body-advertises-2025-06-18.md`, says Claude remote connectors showed "no tools available" because Plug's header advertised `2025-11-25` while the JSON initialize body still advertised `2025-06-18`. `rmcp 1.5.0` changed that premise. In `modelcontextprotocol/rust-sdk` tag `rmcp-v1.5.0`, `crates/rmcp/src/model.rs` defines `ProtocolVersion::V_2025_11_25`, includes it in `KNOWN_VERSIONS`, and sets `ProtocolVersion::LATEST = V_2025_11_25`; `InitializeResult::new(...)` uses `ProtocolVersion::default()`, which is `LATEST`. Source checked locally from https://github.com/modelcontextprotocol/rust-sdk/tree/rmcp-v1.5.0 on 2026-05-17.

### Verdict

**Half-dead.** Delete the manual JSON mutation, not the behavior.

Concrete deletion sequence:

1. In `plug-core/src/http/server.rs:1485`, add typed `with_protocol_version(...)` to `build_initialize_result(...)`, matching `plug-core/src/proxy/mod.rs:3761` and `plug/src/ipc_proxy.rs:520`.
2. Replace the initialize handler's JSON-value rewrite at `plug-core/src/http/server.rs:1119` with the normal typed response path. This likely deletes `json_value_response(...)` / `json_value_response_with_session(...)` if no other caller remains, or restores a typed session response helper.
3. Keep `PROTOCOL_VERSION` / `LATEST_PROTOCOL_VERSION` constants in `plug-core/src/http/server.rs`, `plug-core/src/proxy/mod.rs`, `plug-core/src/server/mod.rs`, and `plug/src/ipc_proxy.rs` until a small shared constant helper is introduced.
4. Keep the regression assertion at `plug-core/src/http/server.rs:2125`.

This can ship as part of Roadmap Do Now #2 before the `rmcp 1.7.0` upgrade. It is not a wire-level change for downstream clients if the response remains `2025-11-25`.
