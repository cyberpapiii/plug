# Tool Naming & Display System Design

**Date**: 2026-03-05
**Status**: Approved

## Problem

211 tools across 10 servers. Current wire names are `server__tool_name` with inconsistent casing (hyphens, camelCase, snake_case mixed). The 98 workspace tools all share one opaque `workspace__` prefix, making it impossible for both humans and LLMs to quickly identify which Google service a tool belongs to.

## Design

### Core Principle

Use MCP's `name`/`title` separation:
- `name` = machine wire name (what the LLM sees and calls)
- `title` = human display name (what clients render in UI)

Both get improved. The LLM gets semantically meaningful prefixes. The human gets pretty formatting.

### 1. Name Sanitization

All upstream tool names are normalized to `snake_case` before any other processing:
- Hyphens to underscores: `create-comment` -> `create_comment`
- camelCase to snake_case: `listProjects` -> `list_projects`, `whoAmI` -> `who_am_i`
- Dots to underscores: `admin.tools.list` -> `admin_tools_list`

This fixes 21 tools today (Notion, Context7, Supermemory, Svelte).

### 2. Server Prefix in `name`

Format: `{Server}__{sanitized_tool_name}`

The server prefix is the **logical service name**, not necessarily the upstream server config name. For most servers, these are the same:

| Config Server | Prefix in `name` | Example |
|--------------|-------------------|---------|
| slack | Slack | `Slack__search_messages` |
| notion | Notion | `Notion__create_comment` |
| krisp | Krisp | `Krisp__search_meetings` |
| imessage | IMessage | `IMessage__get_unread` |
| exa | Exa | `Exa__web_search` |
| context7 | Context7 | `Context7__resolve_library_id` |
| supermemory | Supermemory | `Supermemory__list_projects` |
| svelte | Svelte | `Svelte__get_documentation` |
| supabase | Supabase | `Supabase__execute_sql` |

### 3. Workspace Sub-Service Decomposition

The `workspace` server is special-cased. Its 98 tools are classified into logical sub-services by pattern-matching the tool name, and the prefix becomes the sub-service:

| Sub-Service | Pattern Keywords | Example Tool | Wire Name |
|-------------|-----------------|--------------|-----------|
| Gmail | `gmail` | `get_gmail_messages_content_batch` | `Gmail__get_messages_content_batch` |
| Drive | `drive` | `search_drive_files` | `Drive__search_files` |
| Docs | `doc`, `document`, `table`, `paragraph`, `import_to_google` | `get_doc_content` | `Docs__get_content` |
| Sheets | `sheet`, `spreadsheet`, `conditional_formatting` | `read_sheet_values` | `Sheets__read_values` |
| Slides | `presentation`, `get_page`, `page_thumbnail` | `get_presentation` | `Slides__get_presentation` |
| Calendar | `event`, `calendar`, `freebusy` | `get_events` | `Calendar__get_events` |
| Contacts | `contact` | `search_contacts` | `Contacts__search` |
| Tasks | `task` | `manage_task` | `Tasks__manage` |
| Chat | `message`, `spaces`, `reaction`, `send_message` | `send_message` | `Chat__send_message` |
| Forms | `form` | `create_form` | `Forms__create` |
| Scripts | `script`, `deployment`, `version`, `trigger`, `publish` | `run_script_function` | `Scripts__run_function` |
| Auth | `google_auth` | `start_google_auth` | `Auth__start_google_auth` |

After classification, the sub-service keyword is stripped from the tool name to avoid redundancy (e.g., `get_gmail_messages_content_batch` -> `get_messages_content_batch` under `Gmail`).

**Fallback**: Any workspace tool that doesn't match a pattern keeps `Workspace__` as prefix.

**Config override**: The classification patterns are built-in but can be overridden via `tool_groups` in config per-server. This is a future enhancement, not required for v1.

### 4. Title Generation

Every tool gets a `title` field for human display. Generated by:
1. Take the wire name prefix (e.g., `Gmail`)
2. Add `: `
3. Title-case the tool name portion (underscores become spaces, capitalize each word)

Examples:
| Wire Name (`name`) | Display Name (`title`) |
|--------------------|----------------------|
| `Gmail__get_messages_content_batch` | `Gmail: Get Messages Content Batch` |
| `Slack__search_messages` | `Slack: Search Messages` |
| `Calendar__get_events` | `Calendar: Get Events` |
| `Notion__create_comment` | `Notion: Create Comment` |
| `IMessage__get_unread` | `IMessage: Get Unread` |
| `Supermemory__list_projects` | `Supermemory: List Projects` |

### 5. Deduplication (Strip Redundant Words)

When the sub-service name appears in the tool name, strip it:
- `get_gmail_messages_content_batch` under Gmail -> `get_messages_content_batch`
- `search_drive_files` under Drive -> `search_files`
- `list_presentation_comments` under Slides -> `list_comments`
- `get_doc_content` under Docs -> `get_content`

This keeps wire names short and avoids `Gmail__get_gmail_messages` redundancy.

### 6. Case-Insensitive Routing Fallback

When handling `tools/call`, plug matches the incoming tool name against the routes table. If exact match fails, try case-insensitive lookup. This catches any LLM casing drift (e.g., calling `gmail__get_events` instead of `Gmail__get_events`).

### 7. `_meta` Vendor Metadata

Each proxied tool gets `_meta` with plug-specific grouping data:

```json
{
  "_meta": {
    "com.plug-mcp/upstream_server": "workspace",
    "com.plug-mcp/upstream_tool": "get_gmail_messages_content_batch",
    "com.plug-mcp/service": "gmail"
  }
}
```

This is spec-compliant (vendor-prefixed keys in `_meta`) and enables plug's TUI/CLI to group tools by service without re-parsing names.

### 8. Annotation Enrichment (existing)

Already implemented. Infers `readOnlyHint`/`destructiveHint` from tool name patterns. No changes needed, but now runs on the sanitized+deduplicated name.

## Routing Table

The routing table maps: `wire_name -> (config_server_id, original_upstream_tool_name)`

Example:
```
Gmail__get_messages_content_batch -> ("workspace", "get_gmail_messages_content_batch")
Slack__search_messages            -> ("slack", "conversations_search_messages")
Notion__create_comment            -> ("notion", "create-comment")
Supermemory__list_projects        -> ("supermemory", "listProjects")
```

The original upstream name is preserved exactly for routing `tools/call` back to the correct server.

## What Changes Where

| Component | Change |
|-----------|--------|
| `proxy/mod.rs` `refresh_tools()` | Add sanitization, sub-service classification, title generation, dedup |
| `proxy/mod.rs` `call_tool()` | Add case-insensitive fallback lookup |
| `proxy/mod.rs` `RouterSnapshot` | Store `_meta` and `title` on Tool objects |
| `enrichment.rs` | No changes (runs on sanitized names) |
| `config/mod.rs` | No changes for v1 (built-in patterns, config override is future) |
| `plug/src/main.rs` | Update `plug tools` display to show title when available |
| `daemon.rs` | Pass through title in IPC tool listing |

## Out of Scope

- Config-driven `tool_groups` (future: let users override sub-service classification)
- Tool icons per sub-service
- SEP-1821 dynamic tool discovery (`query` param on `tools/list`)
- Any changes to tool descriptions or inputSchema
