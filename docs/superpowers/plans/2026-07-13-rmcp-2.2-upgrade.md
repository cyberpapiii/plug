# RMCP 2.2.0 Stable Migration Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Upgrade Plug from RMCP 1.7.0 to RMCP 2.2.0 while preserving the stable MCP `2025-11-25` wire contract.

**Architecture:** Treat RMCP 2.2.0 as a Rust API migration, not a protocol redesign. Convert Plug's model construction and access to RMCP's spec-aligned non-exhaustive types at existing boundaries, preserve Plug-owned routing/session behavior, and pin compatibility with focused wire-format tests.

**Tech Stack:** Rust 1.88 MSRV, Cargo workspace, RMCP 2.2.0, Tokio, serde/serde_json, existing Plug unit/integration harness.

## Global Constraints

- Pin RMCP to exactly `2.2.0`.
- Pin `sse-stream` to exactly `0.2.4`; RMCP 2.2.0 calls the plural `from_bytes_stream` API restored in that release.
- Continue negotiating MCP protocol `2025-11-25` upstream and downstream.
- Reject RMCP's known-but-unimplemented `2026-07-28` revision before its server negotiation layer can echo it.
- Preserve stdio, Streamable HTTP, daemon IPC, OAuth, Tasks, elicitation, sampling, resources, prompts, completion, and notification behavior.
- Do not adopt the unreleased July 28 stateless MCP revision or Tasks extension.
- Do not replace Plug's HTTP/session implementation or perform unrelated refactors.
- Keep source-build MSRV at Rust 1.88.

- [ ] **Step 3: Migrate task construction and request parameters**

```rust

    .with_ttl(ttl)
    .with_poll_interval(poll_interval);
```

Use RMCP 2.2 canonical task names in production code; retain old JSON-RPC method strings (`tasks/list`, `tasks/get`, `tasks/result`, `tasks/cancel`) because Plug remains on protocol `2025-11-25`.

- [ ] **Step 4: Migrate cancellation construction and routing**

Outgoing request cancellation:

```rust
CancelledNotificationParam::new(Some(request_id), reason)
```

Incoming anonymous cancellation:

```rust
let Some(request_id) = notification.request_id.clone() else {
    return;
};
```

Do not synthesize an ID and do not broaden cancellation to other active calls.

- [ ] **Step 5: Migrate peer metadata ownership**

Use `Peer::peer_info()` as `Option<Arc<_>>`; use `.as_deref()` only for borrowed helper inputs, or clone individual metadata fields when storing them. Do not retain extra peer/session ownership solely to recover the old borrowed type.
