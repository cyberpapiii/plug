# RMCP 2.2.0 Stable Migration

## Goal

Upgrade Plug from RMCP 1.7.0 to the latest stable RMCP release, 2.2.0, without
changing Plug's public MCP wire contract. Plug will continue to negotiate the
current stable MCP revision, `2025-11-25`.

## Scope

- Pin `rmcp` and `rmcp-macros` through the workspace dependency at 2.2.0.
- Regenerate `Cargo.lock` and review all transitive dependency changes.
- Migrate compile-time API changes in Plug, Plug Core, and the test harness.
- Preserve existing stdio, Streamable HTTP, daemon IPC, OAuth, Tasks,
  elicitation, sampling, resources, prompts, completion, and notification
  behavior.
- Update current-truth and release documentation where it identifies the RMCP
  version or dependency policy.
- Rebuild, sign, and reinstall the local binary after the merged commit passes
  all gates.

## Non-goals

- Do not adopt the unreleased July 28 stateless MCP protocol revision.
- Do not replace Plug's HTTP/session implementation with RMCP's server
  transport as part of this dependency migration.
- Do not redesign the legacy `2025-11-25` Tasks surface or adopt the incompatible
  Tasks extension until its target protocol revision is stable.
- Do not perform unrelated refactors.

## Migration Strategy

First update only the workspace dependency and lockfile, then use the compiler
as the inventory of source-breaking API changes. Resolve each break at the
narrowest existing boundary and preserve the behavior expressed by the current
tests. Any behavioral mismatch discovered during migration gets a focused
regression test before its production fix.

RMCP remains an implementation dependency rather than Plug's product boundary:
Plug's routing, ownership, transport parity, and persistence semantics remain
Plug-owned. New RMCP APIs are adopted only where required for 2.2.0
compatibility or where they are behaviorally identical to the existing path.

## Compatibility Invariants

- Downstream and upstream initialization negotiate `2025-11-25`.
- HTTP requests retain Plug's current protocol-version, session, origin, auth,
  and replay behavior.
- Capability masking remains transport-aware.
- Existing Tasks requests remain compatible with `2025-11-25` clients and
  upstreams.
- Reverse requests, cancellation, progress, subscriptions, and task ownership
  preserve their current routing identities.
- No new configuration or migration step is required for users.

## Verification

The migration is complete only after all of the following pass on the upgraded
lockfile:

- `cargo test --workspace`
- `cargo clippy --workspace --all-targets -- -D warnings`
- `cargo fmt --all -- --check`
- `cargo +1.88.0 check --workspace --all-targets`
- `cargo deny check advisories`
- `scripts/check-todo-status.sh`
- `git diff --check`

After merge and push, `scripts/dev-reinstall.sh` rebuilds and signs the local
binary. The old daemon is replaced, and completion requires one live daemon,
the IPC socket and port 3282 listener, plus recovered client inventory from
`plug status --output json`.

## Rollback

The migration is isolated in conventional commits. If RMCP 2.2.0 introduces an
unacceptable compatibility regression, revert the migration commit and
reinstall the previously pinned 1.7 release line; no user data or configuration
format changes are involved.
